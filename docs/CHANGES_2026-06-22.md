## 1. Risk score rewrite (`risk_score.py`, `hotspots.py`)

### The problem with the old score
The old score **averaged** per-violation severity and vehicle weights across all
violations in a hotspot. Averaging hundreds of violations mean-reverts every
hotspot toward the population centre — the old distribution was **mean 42,
std 7**, nearly constant. A cell with many tankers but mostly scooters averaged
to "medium" and the tanker signal vanished. The score couldn't discriminate.

### The fix — three design changes
1. **Shares, not means.** Instead of "mean severity 56", we use "**62% of this
   cell is main-road parking**" and "**34% are large vehicles**". Shares preserve
   the tail regardless of the scooter background, and are more interpretable.
2. **Small-sample shrinkage (James–Stein toward the global prior).** A cell with
   9 violations that happen to all be main-road parking shows 100% share — that's
   noise, not a hotspot. Each share is shrunk toward the global rate with
   strength `SHRINK_K = 20` (a cell with n=20 is pulled halfway to the prior;
   large cells barely move). This kept small-sample (n<15) cells out of the top ranks.
3. **Multiplicative junction term.** A hotspot at an intersection is
   *categorically* worse, not "+3.75 points worse". The junction flag now
   amplifies: `risk_score = core × (1 + 0.35 × junction_flag)`.

### The formula
```
core = severity_share × 0.45   (% high-impact / main-road parking)
     + density_score  × 0.30   (log-normalised volume)
     + vehicle_share  × 0.25   (% large vehicles, LGV+)

risk_score = core × (1 + 0.35 × junction_flag)   → clamp [0, 100]
```

### Result
- Distribution: **mean 19.6, std 8.3, max 62.8** (was mean 42, std 7).
- Top hotspot **HS-0001 = HAL Old Airport** (681 violations, 85% main-road) —
  genuine high-volume / high-impact, no small-sample artifacts in the top ranks.
- `hotspots.py` now emits the inputs the new score needs (`severity_share`,
  `vehicle_share`, `junction_flag`, `high_impact_count`, `large_vehicle_count`)
  instead of the old mean columns (`severity_score_agg`, `vehicle_score_agg`).
- Parquets regenerated: `hotspots.parquet`, `by_station.parquet`,
  `by_junction.parquet`.

### Downstream impact (verified, no code changes needed)
- **Forecast:** `risk_score` is a static XGBoost feature. Retrained → MAE 5.22,
  still ≤ best naive baseline 5.25. **No regression.**
- **Patrol optimizer:** unchanged. Priority is `risk_score × count`, so the
  shrunk score rescales proportionally; ranking preserved. Both modes verified
  (predictive: 12 assignments, 75% coverage; historical: anchors on HS-0027,
  the highest risk×volume station).

## 2. Station-level forecast (NEW — `/forecast/stations`)

### Why
Per-hotspot weekly counts are **irreducibly noisy** — measured median
coefficient of variation (CV) = **2.0** (std is twice the mean). No feature
fixes that; it's the nature of forecasting 1,196 small cells. Chasing exact
per-hotspot counts is the wrong goal.

Aggregating to the **53 police stations** invokes the law of large numbers:
station-week counts are far more predictable.

### What it delivers (measured on the same clean holdout, W03–W04)
| Metric | Per-hotspot | **Station-level** |
|---|---|---|
| Precision@10 | 0.60 | **0.80** |
| Median CV (noise) | 2.00 | **1.01** |

Station grain roughly **halves** the week-to-week noise and ranks the top
stations better. This is the narrative: *we forecast station trends accurately,
and pair the noisier per-hotspot forecast with the escalation watch — rather
than pretending we can nail exact per-hotspot counts.*

### Implementation
- `forecast.py`: `_build_station_panel`, `_fit_station`, `get_station_forecast`
  (own module-level cache, trains once per server lifetime).
- Uses `reg:squarederror` (not `count:poisson`): station counts reach ~1,900/wk
  and the Poisson objective exponentiates lags, which overflowed.
- **Bug caught & fixed during build:** the first cut predicted off the W06–W13
  data-gap tail (weeks decimated to 26–889 rows by a `validation_status`
  approval backlog), producing absurd +1000% jumps from near-zero baselines.
  Now uses the same **clean-week discipline** as the per-hotspot model
  (predicts from the last clean lag window, W02–W05).
- `analytics.py`: `GET /forecast/stations` endpoint.
- `schemas.py`: `StationForecastResponse`, `StationForecastItem`.
- `frontend_v2`: forecast page now has a "STATION-LEVEL FORECAST" section with
  the CV/accuracy comparison cards + a 53-station table. `api.ts` gains
  `getStationForecast()`.

## 3. Rejected: spatial-lag forecasting (neighbour features)

This was a roadmap todo (`AUDIT_REPORT.md` §8). We **built and A/B-tested it**
rather than ship it blind.

- **Idea:** add neighbour-activity features (sum of same-week counts in the H3
  ring around each hotspot, lagged 1 week) on top of own-lags.
- **Result on the fixed holdout:** MAE 5.405 → 5.334 (−0.07), but
  **Precision@10/20 identical** (0.60 / 0.60). Zero ranking improvement.
- **Decision: rejected.** The CV=2.0 noise ceiling caps the upside; a
  third-decimal MAE gain with no ranking value isn't worth the neighbour-feature
  complexity and per-predict compute. Not integrated.

## 5. SHAP "why" panel (`forecast.py`, `schemas.py`)

### What was built
XGBoost's `pred_contribs=True` (built-in tree-path attribution, equivalent to
SHAP TreeExplainer without the external dependency) is already called at fit
time in `_fit()`. The raw contribution matrix was computed but never surfaced.

**This session wired it through end-to-end:**

- `_top_reasons_from_contribs(contribs_row, feature_names, n=3)` — extracts
  the top-n features by |contribution| magnitude, attaches a human label and
  direction ("up" / "down"). Contributions are in log-space (Poisson objective)
  — magnitude only is exposed, raw values are never shown.
- `shap_reasons: dict[str, list[dict]]` computed once in `_fit()` alongside
  predictions, stored in `_cache`, served per-hotspot at request time —
  zero per-request compute.
- `ForecastItem.top_reasons: list[ReasonItem]` — already in the schema
  (`ReasonItem`: `feature`, `label`, `direction`).
- Reasons suppressed for hotspots with `predicted_count < 1.0` (all-zero lags,
  no signal).

**Frontend:** the forecast table already renders the `top_reasons` chips as
"4-week trend ↑", "Last week's volume ↑", etc.

### The `risk_score` tautology fix

After shipping, the top-3 chips on almost every hotspot showed:
```
4-week trend ↑  |  Last week's volume ↑  |  Static risk score ↑
```

The third chip is circular — `risk_score` is derived from historical violations,
so "predicted high because risk score is high" tells the operator nothing new.
It reads as the model explaining itself with itself.

**Fix:** `_top_reasons_from_contribs` now skips `risk_score` when building the
shortlist, pulling the next-ranked feature instead (typically "Monthly pattern"
or "Junction proximity"). The feature remains in `_FEATURE_COLS` and
contributes to model accuracy — it is only excluded from the *display*.

```python
# Before — top 3 including risk_score
for feat, val in pairs[:n]:
    out.append(...)

# After — skip risk_score, pull next in rank
for feat, val in pairs:
    if feat == "risk_score":
        continue
    out.append(...)
    if len(out) == n:
        break
```

This is a display-layer filter, not a model change. MAE and Precision@N are
unchanged.

## 6. Shift-level forecasting (`forecast.py`, `schemas.py`)

### The problem it solves
Yesterday's station-level forecast tells a commander "Indiranagar will have
~320 violations next week." That's a headcount; it doesn't tell them *when* to
deploy. Shift-level forecasting answers the operational question:
**"Which day and shift should we staff up?"**

### Implementation
All the data was already present — it just needed combining:

1. **Volume source:** `get_station_forecast()` — next-week predicted count per
   station (already computed, cached).
2. **Temporal signature:** `temporal.parquet` — `(hotspot_id, hour, day_of_week,
   count)` cells recording each hotspot's historical hour×day pattern.
3. **Distribution logic** (`_compute_peak_shifts` in `forecast.py`):
   - Each hotspot in the station gets its historical `violation_count` share of
     the station's predicted total (`hotspot_predicted = station_predicted × share`).
   - For hotspots with ≥5 temporal cells: distribute `hotspot_predicted`
     proportionally across their `(day_of_week, shift)` signature.
   - For sparse hotspots (<5 cells): distribute uniformly across all 21 windows
     (safe fallback — no signal to exploit).
   - Hours are bucketed into three named shifts: **Morning** (06–13),
     **Afternoon** (14–21), **Night** (22–05).
   - Aggregate across hotspots → 21 `(day, shift)` buckets → return top 3.
   - Invariant enforced: all 21 windows sum to 100% ± 0.5% (assert guard).

4. **Output shape** — `StationForecastItem.peak_shifts: list[PeakShift]`:
   ```json
   [
     {"day": "Tue", "shift": "Morning",   "pct": 18.4},
     {"day": "Wed", "shift": "Afternoon", "pct": 15.2},
     {"day": "Mon", "shift": "Morning",   "pct": 12.7}
   ]
   ```

5. **Schema:** `PeakShift(day, shift, pct)` + `StationForecastItem.peak_shifts`
   already reflected in `schemas.py` from yesterday's station-forecast session —
   no schema changes needed today.

### Design constraint
The `time_window` field on `PatrolAssignment` in the patrol optimizer is derived
from `_format_peak_window` (a historical heuristic). Shift-level outputs are
surfaced on the station forecast page rather than wired into the patrol
assignment directly — that coupling is the logical next step but was out of
scope for this session.

## 7. POI metro stat card (`PoiClient.tsx`)

### What
A single headline stat card above the category table on the POI Spillover page:

> **X% of tagged violations at metro-proximate hotspots**

### Implementation — zero backend changes
The existing `GET /poi-stats` response already returns `by_category[]` with
`total_violations` per category. The metric is computed entirely on the client:

```typescript
const metroCategory = stats.find((c) => c.poi_category === "metro");
const totalViolations = stats.reduce((sum, c) => sum + c.total_violations, 0);
const metroShare =
  metroCategory && totalViolations > 0
    ? ((metroCategory.total_violations / totalViolations) * 100).toFixed(1)
    : null;
```

Rendered as a `border border-border p-6` KPI card (matching the offenders page
pattern) with a `Train` icon from lucide-react. The card is conditionally
rendered — if `metroShare` is null (no metro category in data), nothing shows.

### Wording rationale
The stat says "metro-proximate" not "within 200m". The POI tagging is
keyword-based on location text (`poi_tagging.py`), not a radius query against
an external geocoded database. "Metro-proximate" is literally what the data
says and is defensible under scrutiny. "Within 200m" would be fabricated.

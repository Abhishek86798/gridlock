## P0 — Will break during demo (fix immediately)

### ~~P0-1~~ · ~~`assert` in hot path crashes `/forecast/stations` with 500~~ ✅ FIXED
**File:** `backend/app/services/forecast.py:785–787`

```python
assert 99.5 <= all_pcts <= 100.5, (
    f"Station {station}: 21-window pct sum {all_pcts:.3f}% outside [99.5, 100.5]"
)
```

`_compute_peak_shifts()` is called 53 times per `/forecast/stations` request (once per
station). If floating-point accumulation in the bucket loop lands outside `[99.5, 100.5]`
for any station — which can happen with sparse hotspot data — Python raises `AssertionError`
uncaught. FastAPI propagates this as an unhandled 500, taking down the station forecast
endpoint and with it the station-shift section on the Deploy page and the station table on
the Forecast page simultaneously.

**Fix:**
```python
# Replace the assert with:
if not (99.5 <= all_pcts <= 100.5):
    return []   # fallback: no shift data for this station
```
**Estimated effort:** 2 minutes

### ~~P0-2~~ · ~~Temporal page fetch has no `.catch()` — hangs indefinitely on network failure~~ ✅ FIXED
**File:** `frontend_v2/src/app/temporal/page.tsx:26–32`

```ts
fetch(`${BASE_URL}/temporal/${selectedId}`)
  .then(res => res.json())
  .then(data => {
    setMatrix(data.matrix || []);
    setLoading(false);
  });
// No .catch() — setLoading(false) never fires on network error
```

If the backend is down or a CORS error occurs during the demo, `setLoading(false)` is never
called. The page stays in the loading overlay indefinitely with no visible error.

**Fix:**
```ts
fetch(`${BASE_URL}/temporal/${selectedId}`)
  .then(res => res.ok ? res.json() : { matrix: [] })
  .then(data => { setMatrix(data.matrix || []); setLoading(false); })
  .catch(() => setLoading(false));
```
**Estimated effort:** 3 minutes

### ~~P0-3~~ · ~~`get_station_forecast()` recomputes 178,080 Python row iterations on every request~~ ✅ FIXED (QW-5)
**File:** `backend/app/services/forecast.py:951–965`, `forecast.py:755–777`

```python
for _, row in hs_in_station.iterrows():      # 20 hotspots/station
    ...
    for _, tc in hs_temporal.iterrows():     # 168 temporal cells/hotspot
        shift = _hour_to_shift(int(tc["hour"]))
        ...
```

`_compute_peak_shifts()` is called 53 times (once per station) inside
`get_station_forecast()`. Each call runs a Python-level `iterrows()` over a station's
hotspots, with a nested `iterrows()` over that hotspot's temporal cells. Worst case:
20 hotspots × 168 cells × 53 stations = **178,080 Python-level row iterations per
`/forecast/stations` request**. The model fit is cached in `_station_cache`, but the
shift-window computation is not — it re-runs on every call.

**Fix:** Cache the final `items` list inside `_station_cache`. In `_fit_station()`, after
building `snap_all`, compute the full station forecast items once and store them:

```python
# In _fit_station(), replace the prediction-only return with a full items computation,
# or add a module-level _station_items_cache:

_station_items_cache: list | None = None

def get_station_forecast() -> dict[str, Any]:
    global _station_items_cache
    ctx = _ensure_station_trained()
    if _station_items_cache is not None:
        return {**ctx_meta, "forecast": _station_items_cache}
    # ... build items ...
    _station_items_cache = items
    return result
```

**Estimated effort:** 15 minutes

### ~~P1-2~~ · ~~Deploy page `Promise.all` has no `.catch()` — hangs on network failure~~ ✅ FIXED
**File:** `frontend_v2/src/app/deploy/page.tsx:20–28`

```ts
Promise.all([getPatrol(units), getStationForecast()]).then(([patrol, stationForecast]) => {
  setData(patrol);
  ...
  setLoading(false);
});
// No .catch() — loading overlay never clears on error
```

A network-level failure (not an API error — those are caught in `fetchApi`) will reject the
`Promise.all` and `setLoading(false)` is never called. The Deploy page shows "Calculating
optimal deployment routes..." indefinitely.

**Fix:**
```ts
Promise.all([getPatrol(units), getStationForecast()])
  .then(([patrol, stationForecast]) => {
    setData(patrol);
    const lookup: Record<string, any[]> = {};
    for (const row of stationForecast.forecast || []) {
      if (row.peak_shifts?.length) lookup[row.police_station] = row.peak_shifts;
    }
    setPeakShiftByStation(lookup);
    setLoading(false);
  })
  .catch(() => setLoading(false));
```
**Estimated effort:** 2 minutes

### P1-3 · `/priority` endpoint calls `_tier()` closure per-row inside `iterrows()`
**File:** `backend/app/routers/hotspots.py:115–132`

```python
items = [
    {
        ...
        "priority_tier": _tier(float(row["risk_score"])),
    }
    for i, (_, row) in enumerate(df.iterrows())   # up to 500 rows
]
```

`p90` and `p99` are correctly computed once before the loop. The per-row Python closure
call overhead is acceptable at the default `limit=50`, but with `limit=500` (max) this
becomes measurable.

**Fix:**
```python
import numpy as np
thresholds = [-np.inf, p90, p99, np.inf]
labels = ["Standard", "Elevated", "Critical"]
df["_tier"] = pd.cut(df["risk_score"], bins=thresholds, labels=labels).astype(str)
items = df[["hotspot_id","risk_score","logging_window","police_station","_tier"]] \
    .rename(columns={"_tier":"priority_tier"}) \
    .reset_index(drop=True) \
    .to_dict(orient="records")
# then add rank manually
```
**Estimated effort:** 10 minutes

## P2 — Technical debt worth noting (fix after demo)

### P2-1 · Thread safety of `_cache` and `_station_cache` singletons
**File:** `backend/app/services/forecast.py:122`, `forecast.py:810`

```python
_cache: dict[str, Any] | None = None

def _ensure_trained() -> dict[str, Any]:
    global _cache
    if _cache is None:
        _cache = _fit(...)
```

Under uvicorn's default single-worker + asyncio model, the GIL and single-process event
loop make this safe for the demo (startup warmup eliminates the race window entirely).
Under `uvicorn --workers N` (multi-process), each process gets its own `_cache` —
multiplying memory usage N-fold. Not a demo risk; flag for production deployment.

### P2-2 · `enforcement-quality` endpoint uses Python-level groupby iteration
**File:** `backend/app/routers/analytics.py:262–271`

```python
grp = vdf.groupby("police_station")
for station, g in grp:
    total = len(g)
    rejected = (g["validation_status"] == "rejected").sum()
    ...
```

Fine for 53 stations. Vectorised alternative for future scale:
```python
total = vdf.groupby("police_station").size()
rejected = (vdf[vdf["validation_status"] == "rejected"]
            .groupby("police_station").size()
            .reindex(total.index, fill_value=0))
rows = (rejected / total.clip(lower=1)).round(4).reset_index()
```

### P2-4 · `store.*` DataFrames not always `.copy()`-d before filter assignment
**File:** `backend/app/routers/analytics.py:53–56`, `hotspots.py:71`

```python
df = store.by_station        # no .copy()
if min_hotspots > 1:
    df = df[df["hotspot_count"] >= min_hotspots]   # may be a view
```

Pandas boolean indexing and `.head()` return views in some versions. A future in-place
operation would mutate the shared store. The patrol optimizer correctly calls
`df = store.hotspots.copy()` (patrol_optimizer.py:66). Routers should match:
```python
df = store.by_station.copy()
```

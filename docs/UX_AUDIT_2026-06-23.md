# Trinetra UX Audit — Pre-Demo Report
**Date:** 2026-06-23 | **Demo:** 3 July 2026 | **Audience:** BTP leadership + Flipkart engineering judges

---

## Page-by-Page Findings

---

### 1. Overview (`page.tsx`)

**Self-explanatory score: NEEDS WORK**

A police commander opens this page and sees four numbers: "169,432", "1,196", "Koramangala", "94.2%". None of them have enough context to act on. The table below shows hotspot IDs like "HS-0001" with a "Risk" column showing "62.8" — no scale, no unit, no explanation of what to do next.

**Issue 1 — KPI tiles have no units or scale context** (`page.tsx:44–49`)

"1,196" next to "Hotspot Zones" tells a commander nothing. Is that all of Bengaluru? Is 1,196 high? The "Afternoon Blind Spot" tile shows "94.2%" with no explanation of what a "blind spot" is or why 94% is alarming.

Fix — change the KpiCard subtitle text (add a `subtitle` prop to `KpiCard` at line 100):
- Total Violations: add subtitle `"recorded violations in dataset"`
- Hotspot Zones: add subtitle `"geo-clusters across Bengaluru"`
- Busiest Station: add subtitle `"by violation volume"`
- Afternoon Blind Spot: add subtitle `"hotspots with no patrol 12–17 hrs"` and color the value red since 94% is a problem

**Issue 2 — Risk score column has no scale** (`page.tsx:83`)

The table shows `{row.risk_score?.toFixed(1)}` as a bare number (e.g. "62.8"). The header just says "Risk". A commander has no idea if 62.8 is dangerous or routine.

Fix — change the column header at line 65 from `"Risk"` to `"Risk Score (0–100)"`. Add a one-line note below the table:
> *Risk score: 0–100. Scores above 70 are high-priority zones where violations cluster most densely.*

**Issue 3 — No "start here" CTA; the page is a dead end** (`page.tsx:51–96`)

After viewing the priority queue, there is nothing pointing the commander to the next logical action. The page ends after the table.

Fix — add a banner below the table:
```tsx
<div className="border border-border p-6 flex items-center justify-between">
  <div>
    <p className="text-sm text-text-primary font-light">
      Ready to act on these hotspots?
    </p>
    <p className="text-xs text-text-secondary mt-1">
      Use Patrol Deployment to generate an optimised unit roster for next week.
    </p>
  </div>
  <Link href="/deploy" className="text-xs uppercase tracking-widest px-6 py-3 border border-border hover:bg-bg-elevated transition-colors">
    Go to Patrol Deployment →
  </Link>
</div>
```

Also: the `TierBadge` at line 114 uses white-on-white for "Critical" (`text-text-primary border border-text-primary`). On a light laptop screen this is unreadable. Critical should be `text-red-400 border-red-500/50 bg-red-900/20`.

**Issue 4 — "Peak Block" column shows values like "10:00–14:00" with no explanation** (`page.tsx:84`)

Fix — rename `"Peak Block"` to `"Peak Hours"` and add a column header tooltip or footnote: *"Hours when this zone typically sees the highest violation rate."*

### 2. Live Map (`map/page.tsx` + `LiveMap.tsx`)

**Self-explanatory score: NEEDS WORK**

The map renders with no title, no explanation panel, and no legend except the Risk Tier box in the bottom-right. A commander opening this page sees colored dots and has no idea what the circle radius means, what "Vio" means in the popup, or what "Win" means.

**Issue 1 — Map page has no title or purpose statement** (`map/page.tsx:21–28`)

The entire `MapPage` component returns a full-screen map with zero text. The page title in the sidebar says "Live Map" but the page itself has no header.

Fix — add a floating info bar at the top of the map, overlaid like the search bar:
```tsx
<div className="absolute top-6 right-6 z-[1000] bg-black/80 backdrop-blur border border-border px-4 py-2 max-w-xs">
  <p className="text-[10px] text-text-secondary uppercase tracking-widest">Live Hotspot Map</p>
  <p className="text-xs text-text-secondary font-light mt-1">Circle size = risk score. Click any dot for details.</p>
</div>
```

**Issue 2 — Popup labels are cryptic abbreviations** (`LiveMap.tsx:187–197`)

The popup shows "Vio", "Win", "POI" as labels. A police commander will not know these mean Violations, Time Window, Point of Interest.

Fix — in `LiveMap.tsx` popup section, change:
- Line 191: `"Vio"` → `"Violations"`
- Line 195: `"Win"` → `"Peak Hours"`
- Line 200: `"POI"` → `"Near"`

**Issue 3 — Legend doesn't explain what the circle SIZE means** (`LiveMap.tsx:289–300`)

The legend explains color tiers but `radius` is `Math.max(5, hs.risk_score / 8)` (line 128) — larger circles = higher risk. This is invisible to a user.

Fix — add to the legend block:
```tsx
<div className="border-t border-border mt-4 pt-4 text-[10px] text-text-secondary tracking-wide font-light">
  Circle size also reflects risk — larger = higher score
</div>
```

**Issue 4 — No instructions for patrol overlay** (`map/page.tsx:17–18`)

Patrol routes appear when `units > 0` is passed as a URL param, but there is no UI on the map page itself to trigger this. A commander would never discover this feature.

Fix — add a one-line hint under the search bar on the map: *"Adjust 'Patrol Units' in the sidebar to overlay deployment routes."*

### 3. Forecasts (`forecast/page.tsx`)

**Self-explanatory score: NEEDS WORK**

This is the most data-dense page. The KPI tiles use jargon ("Model MAE"), the escalation banner tells you something is wrong but not what to do, the SHAP "Why" column has no legend, and the station-level section shows "CV 1.01 vs 2.00" with no plain-English translation.

**Issue 1 — "Model MAE" KPI tile is meaningless to a non-technical user** (`forecast/page.tsx:68–71`)

The tile says "Model MAE" with value "5.3" and subtitle "violations / hotspot". A police commander does not know what MAE is.

Fix — rename the tile label and subtitle:
- Label: `"Forecast Accuracy"` (was `"Model MAE"`)
- Value: keep `5.3`
- Subtitle: `"avg error ±5 violations per hotspot/week — within normal variance"`

**Issue 2 — Escalation banner has no "what to do" action** (`forecast/page.tsx:90–105`)

The banner says *"Action Required: X hotspots are predicted to surge >20%."* but the only button is "View Alerts" which opens a modal that also offers no next step. The modal closes and the user is stuck.

Fix — in the alert modal footer at line 133, add a navigation button alongside "Close":
```tsx
<Link
  href="/deploy"
  onClick={() => setIsAlertModalOpen(false)}
  className="text-sm font-medium text-text-primary bg-[#EF4444]/10 hover:bg-[#EF4444]/20 px-6 py-2 rounded transition-colors mr-3"
>
  Go to Patrol Deployment →
</Link>
```
Also change the banner text from `"Action Required"` to `"Action Required — reassign patrol units"` and add a direct `<Link href="/deploy">` alongside "View Alerts".

**Issue 3 — SHAP "Why" chips have no legend or tooltip** (`forecast/page.tsx:208–220`)

The table column is labeled "Why" and chips show text like "4-week trend ↑" or "recent spike ↑" in red/grey. A commander has no idea these are model explanations or what the arrows mean.

Fix — change the column header at line 179 from `"Why"` to `"Why it's rising"`. Add a `title` tooltip on the `<th>`:
```
title="The model's top reason this hotspot is forecast to spike — based on its recent trend, surrounding cluster, and seasonal patterns"
```
Also add a footnote below the table:
> *"↑ = factor is pushing violations higher this week. ↓ = factor is stabilising the hotspot."*

**Issue 4 — Bar chart has no Y-axis label and tooltip shows raw field names** (`forecast/page.tsx:151–164`)

The chart `YAxis` at line 153 has no `label`.

Fix:
- Add `label={{ value: 'Violations / week', angle: -90, position: 'insideLeft', fill: '#9CA3AF', fontSize: 11 }}` to `YAxis`
- The `name` props on `Bar` (line 160–161) are already good: "Predicted (XGBoost)" and "Baseline (W01–W05 avg)" — keep them

**Issue 5 — Station-level section: "CV" and "P@10" are unexplained statistics** (`forecast/page.tsx:296–333`)

The card at line 297 says "Top-10 Ranking Accuracy" and shows "0.80 station vs 0.62 hotspot". The value "0.80" reads as a decimal, not a percentage. The card at line 313 says "Week-to-Week Noise (CV)" with "1.01 vs 2.00".

Fix:
- **P@10 card (line 300–305):** Display as percentage: `{(stationData.precision_at[10] * 100).toFixed(0)}%` → shows "80% station vs 62% hotspot". Change subtitle to `"8 out of every 10 top-ranked stations are genuinely high-risk zones"`.
- **CV card (line 316–320):** Change subtitle to `"Station counts swing ±101% week-to-week vs ±200% for individual hotspots — far easier to plan around"`.
- Change tile label `"Week-to-Week Noise (CV)"` to `"Forecast Stability"`.

### 4. Patrol Deployment (`deploy/page.tsx`)

**Self-explanatory score: CLEAR** (best page in the dashboard, but has fixable gaps)

The deployment page is the strongest in the dashboard. The header copy is good. The escalation watch cards are self-contained. But the coverage curve chart is unlabeled, and a commander reading "75.3% spatial risk-weighted" will not know if that's good.

**Issue 1 — Coverage curve chart has no axis labels and the current point is unexplained** (`deploy/page.tsx:107–122`)

The `XAxis` shows numbers (unit counts) and `YAxis` shows 0–100 but neither has a label. The green dot (`ReferenceDot`) appears but there's no annotation explaining "this is you."

Fix:
- Add `label={{ value: 'Patrol Units', position: 'insideBottom', offset: -5, fill: '#9CA3AF', fontSize: 11 }}` to `XAxis`
- Add `label={{ value: 'Hotspot Coverage %', angle: -90, position: 'insideLeft', fill: '#9CA3AF', fontSize: 11 }}` to `YAxis`
- Add a caption below the chart: `"Each additional unit covers fewer new hotspots — the curve flattens as coverage saturates. The green dot marks your current deployment."`

**Issue 2 — "Priority Coverage 75.3%" has no context for whether it's adequate** (`deploy/page.tsx:99–102`)

The subtitle says "spatial risk-weighted" which is jargon.

Fix — change the card subtitle at line 101 from `"spatial risk-weighted"` to:
`"of high-risk zones have a patrol unit assigned — {(100 - covPct).toFixed(1)}% remain uncovered"`

Add conditional color: `covPct < 60` → `text-critical`, `60–80` → `text-amber-400`, `>80` → `text-patrol`.

**Issue 3 — Assignments Roster table has no context for "Risk Score" column** (`deploy/page.tsx:228`)

Fix — change column header to `"Risk Score (0–100)"`.

**Issue 4 — No "mode toggle" UI exists despite being referenced in the header**

The header copy mentions units are allocated against "predicted load" but there's no way to compare against historical allocation. A judge asking "can I see what historical deployment looked like?" gets no answer.

Fix — add a disabled comparison button in the header area:
```tsx
<button disabled className="text-[10px] uppercase tracking-widest opacity-40 cursor-not-allowed px-4 py-2 border border-border">
  Historical comparison — coming soon
</button>
```

### 6. Temporal Distribution (`temporal/page.tsx`)

**Self-explanatory score: CONFUSING**

This page has the most severe UX problems. A 24×7 table of numbers with no explanation of what "count" means, no color scale legend, no peak window callout, and no action guidance. The title is truncated on screen ("TEMPORAL DIST").

**Issue 1 — Title is truncated and page purpose is unclear** (`temporal/page.tsx:51`)

Fix — change title to `"VIOLATION TIMING"`. Change subtitle to:
> *"Select a hotspot to see which hours and days see the most violations. Darker red cells = more violations. Use this to set patrol shift timings."*

**Issue 2 — No color scale legend anywhere on the heatmap** (`temporal/page.tsx:71–109`)

The heatmap uses `rgba(239, 68, 68, intensity)` for coloring but nowhere is it explained that darker = more violations.

Fix — add a color scale legend above the heatmap:
```tsx
<div className="flex items-center gap-4 mb-4 text-[10px] text-text-secondary">
  <span>Fewer violations</span>
  <div className="flex gap-0.5">
    {[0.1, 0.3, 0.5, 0.7, 0.9].map(v => (
      <div key={v} className="w-6 h-4" style={{ backgroundColor: `rgba(239,68,68,${v * 0.8})` }} />
    ))}
  </div>
  <span>More violations</span>
</div>
```

**Issue 3 — No peak window callout** (`temporal/page.tsx` — absent)

The heatmap shows 168 cells. A commander cannot visually extract "the top 3 time slots."

Fix — add a "Peak Windows" summary computed from the matrix data, above the heatmap:
```tsx
const flatCells = matrix
  .sort((a, b) => b.count - a.count)
  .slice(0, 3)
  .map(c => `${days[c.day_of_week]} ${c.hour.toString().padStart(2,'0')}:00–${(c.hour+1).toString().padStart(2,'0')}:00`);

// Render:
<div className="flex gap-4 mb-6">
  <span className="text-[10px] text-text-secondary uppercase tracking-widest">Peak windows:</span>
  {flatCells.map((label, i) => (
    <span key={i} className="text-[10px] font-light text-critical bg-critical/10 px-3 py-1">{label}</span>
  ))}
</div>
```

**Issue 4 — No action CTA** (`temporal/page.tsx` — absent)

Fix — add below the heatmap:
> *"Use peak hours from this hotspot to set patrol shift windows on the Patrol Deployment page."*

## Cross-Cutting Issues

### CC-1 — Risk score (0–100) is never explained anywhere in the UI

`risk_score` appears in Overview (table column), Map (popup), Stations (table column), Junctions (table column), and Patrol Deployment (roster table). In none of these places is the 0–100 scale described.

**Fix:** Add a consistent footnote wherever `risk_score` appears:
> *"Risk Score: 0–100. Combines violation density, frequency, and cluster size. Above 70 = high priority."*

Best placements: Overview page (entry point) and Stations page.

## Demo Flow Gaps

The judge follows: **Overview → Live Map → Forecasts → Patrol Deployment → POI → Repeat Offenders**

**Gap 1: Overview → Live Map (no bridge)**
A judge sees the priority table and has no prompt to go to the map. Fix: add the "start here" CTA (Overview Issue 3) with a secondary link to the Live Map.

**Gap 2: Live Map → Forecasts (no bridge)**
The map page is full-screen with no text and no visible next step. Fix: add the floating info overlay with text: *"For predicted hotspot surges next week → Forecasts"*.

**Gap 3: Forecasts — "why is this hotspot ranked #1?"**
The SHAP chips are in the table but the "Why" column is the 8th column. On a 1280px laptop with the sidebar open (~1008px usable width), the "Why" column is likely truncated off-screen.

Fix — move the "Why" column to column position 2 (immediately after "Hotspot"), or add a click-to-expand drawer showing the full SHAP breakdown per row.

**Gap 4: Forecasts → Patrol Deployment (no bridge)**
The escalation banner opens a modal. The modal has no path to Patrol Deployment. A judge reads the modal and is stuck.

Fix: modal footer "Go to Patrol Deployment →" link (Forecast Issue 2 fix above).

**Gap 5: Patrol Deployment — "what does 75% coverage mean?"**
The subtitle says "spatial risk-weighted" — sounds technical. Fix: revised subtitle from Deploy Issue 2 that states the uncovered percentage explicitly.

**Gap 6: Patrol Deployment — units slider is in the sidebar, not on the page**
A judge wants to increase units to improve coverage but the control is in the sidebar with no visible connection to the KPI tiles.

Fix: add above the KPI cards: *"Adjust Patrol Units in the left sidebar to recalculate coverage."*

**Gap 7: POI Spillover — "so what?"**
After seeing metro stations cause 23% of violations, a judge asks "what should I do?" The page has no answer. Fix: add action guidance from POI Issue 4.

**Gap 8: Repeat Offenders — "can we act on this?"**
The "Pin" button is confusing. "Pin HS-0064" reads like a UI action, not a police action.

Fix: rename to `"Track on Map"` / `"Tracking ✓"` (Offenders Issue 4).

## Summary Verdict

The **Patrol Deployment** page is production-ready. The **Forecasts** page is strong but fails the "what do I do now?" test. The **Temporal Distribution** and **POI Spillover** pages will confuse a first-time commander without the copy fixes.

The single biggest demo risk is **F2 (no CTA from Forecasts to Patrol Deployment)** — a judge who sees escalating hotspots and can't find the action will conclude the system is surveillance-only. The second biggest is **F1 (risk score never explained)** — it appears on every page and means nothing without the 0–100 scale.
